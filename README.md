# Mini Rippling

**A student-built backend clone of the HR platform Rippling.**

Mini Rippling is essentially HR written in code. It handles the core responsibilities of a real HR system — onboarding employees and assigning them a unique ID, tracking attendance, managing leaves, and running payroll. The goal behind building this was simple: to demonstrate solid backend development skills and showcase hands-on experience with real-world technologies.

Together they form a stack that mirrors what you'd find in production systems at modern tech companies.

## 🧩 Modules

### 👤 Employee
The Employee module is the backbone of the entire system. Every employee gets a unique auto-generated UUID as their internal ID, plus a human-readable `employeeCode` (like `EMP-0001`) for day-to-day use. The module supports a self-referencing manager hierarchy — meaning any employee can be a manager of other employees — because real offices have project leads, associate managers, and department heads at different levels. Employee credentials are intentionally kept in a separate `UserAccount` entity rather than on the Employee itself, because an employee record exists the moment they are hired, but a login account is only created when they actually need system access.

### 🏢 Department
Each department has a name, description, and a designated head — who is also just an Employee in the system. This was a conscious decision: a department head doesn't get a special entity, they start as a regular employee and get assigned the head role. Departments are kept intentionally lean for now, with room to expand in future versions.

### 📅 Attendance
The Attendance module tracks daily clock-in and clock-out for every employee. A unique constraint is enforced at the **database level** (not just application level) to guarantee exactly one record per employee per day — this cannot be bypassed or faked regardless of how the application behaves. An **idempotency key** is generated by the client before sending a punch request — if the network fails and the request is retried, the same key arrives and the duplicate is safely rejected. Late minutes and overtime minutes are calculated per record. HR or admins can manually correct wrong records, and every correction is audit-logged — recording who made the change, when, and why — so employees always know if their record was touched and by whom.

### 📋 Leave
Employees can apply for six types of leave — Casual, Sick, Earned, Unpaid, Maternity, and Paternity — covering the most common real-world HR scenarios. When calculating `totalDays`, weekends are excluded because not every month has the same number of working days and it wouldn't be fair to count a Saturday as a leave day. The leave goes through a full approval workflow: it starts as `PENDING`, then moves to `APPROVED` or `REJECTED`. Every approval and rejection is timestamped and the rejecting manager must provide a mandatory reason. A future version will include public holiday calendars and potentially a calendar integration.

### 💼 Employment History
Every role change, salary revision, or promotion creates a **new immutable record** rather than updating the existing one. This is intentional — employment history is a factual record of what happened, not something that should be rewritten. The `effectiveFrom` and `effectiveTo` columns version each record in time, so the system always knows what an employee's salary and role were at any given point. Salary is stored as `BigDecimal` — never `float` or `double` — because floating point rounding errors in financial calculations can cause real money to vanish at scale.

### 💵 Payroll
The payroll system is split into two entities — `PayrollRun` and `PayrollEntry` — deliberately. A `PayrollRun` represents the initiation of a payroll cycle for a given period (who triggered it, when, for what date range). A `PayrollEntry` is the actual per-employee calculation that happens inside that run — base salary, days worked, leave deductions, overtime, bonus, and net pay. Every entry stores a full `breakdownJson` — a JSON snapshot of how every rupee was calculated — so the payroll is fully auditable and traceable. An idempotency key on `PayrollRun` prevents the same payroll period from being processed twice — if a retry is attempted, it gets rejected rather than crediting employees a second time.

## 🚀 Getting Started

### Prerequisites

- Java 17+
- Maven 3.8+
- PostgreSQL 14+

### 1. Clone the repository

```bash
git clone https://github.com/AkshayTeja3/mini-rippling.git
cd mini-rippling
```

### 2. Create the database

```sql
CREATE DATABASE mini_rippling;
```

### 3. Configure application properties

Edit `src/main/resources/application.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/mini_rippling
spring.datasource.username=your_postgres_username
spring.datasource.password=your_postgres_password

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
```

### 4. Run the application

```bash
./mvnw spring-boot:run
```

### 5. Test with the seeded admin account

On first startup, `DataInitializer` automatically seeds the database with a default admin account:

```
Email:    teja@company.com
Password: password123
```

Use these credentials to log in and get a JWT token to test all endpoints.

## 🔐 Authentication Flow

Mini Rippling uses **stateless JWT authentication**. No sessions are stored on the server — every request must carry a valid token.

```
1. HR hires employee        →  POST /api/employees/hire
2. Employee registers       →  POST /api/auth/register  (with employeeId + password)
3. Employee logs in         →  POST /api/auth/login     (returns JWT token)
4. Use token on all calls   →  Authorization: Bearer <token>
```

Tokens are signed with **HMAC-SHA256**, embed the employee UUID and email as claims, and expire after **24 hours**. Every incoming request passes through `JwtFilter` which validates the token and sets the authenticated employee into Spring Security's context before the request reaches any controller.

Only `/api/auth/**` endpoints are public. Every other endpoint requires a valid JWT.

## 💵 Payroll Engine

The payroll engine processes all active employees for a given period in a single run.

**Calculation formula per employee:**

```
Daily Rate      =  baseSalary ÷ totalWorkingDays
Earnings        =  dailyRate × daysWorked
Deductions      =  dailyRate × unpaidLeaveDays
Overtime Pay    =  (dailyRate ÷ 8) × overtimeHours
Net Pay         =  earnings - deductions + overtimePay
```

- All values use `BigDecimal` with `HALF_UP` rounding — never `float` or `double`
- Salary is sourced from `EmploymentHistory` — the record where `effectiveTo` is null (currently active)
- Weekends are excluded from `totalWorkingDays`
- A full `breakdownJson` is stored per employee entry — every rupee is traceable
- If one employee's calculation fails, their entry is skipped and logged — the rest of the run continues uninterrupted
- An idempotency key (`RUNTYPE-YEAR-MONTH`) prevents the same period from being processed twice

## ⚠️ Exception Handling

All exceptions are handled centrally by `GlobalExceptionHandler` using `@RestControllerAdvice`. Every error returns a consistent JSON structure:

```json
{
  "timestamp": "2026-03-22T15:30:00",
  "status": 404,
  "error": "Employee not found",
  "path": "/api/employees/123"
}
```

| Condition | HTTP Status |
|---|---|
| Resource not found | `404 NOT FOUND` |
| Duplicate / already exists | `409 CONFLICT` |
| Invalid credentials / disabled account | `401 UNAUTHORIZED` |
| Invalid state (already approved etc.) | `400 BAD REQUEST` |
| Everything else | `500 INTERNAL SERVER ERROR` |

## 🔧 Known Limitations & Planned Improvements

- **HireEmployee hardcoded values** — `role` and `salary` from the request are currently ignored and defaulted to `SDE_1` and ₹500,000 in `EmploymentHistory`. To be fixed in refactoring.
- **Employee code generation** — uses `count + 1` which could produce duplicate codes if employee records are deleted. A database sequence would be safer.
- **Shift timings hardcoded** — `9:00 AM` start and `6:00 PM` end are fixed. Configurable shift timings per department or employee is a planned improvement.
- **Exception mapping by string** — `GlobalExceptionHandler` maps HTTP status codes by checking error message keywords. Replacing with custom exception classes (`ResourceNotFoundException`, `ConflictException` etc.) would be more reliable.
- **JWT secret hardcoded** — must be moved to environment variables before any production use.
- **No role-based access control (RBAC)** — all authenticated users can currently access all endpoints. Restricting payroll and hiring to admin/HR roles only is a planned improvement.
- **Bean Validation gaps** — `@NotNull`, `@Email`, `@NotBlank` annotations not yet applied comprehensively across all DTOs.
- **Holiday calendar** — leave `totalDays` currently excludes weekends only. Public holiday support and calendar integration planned for a future version.
- **Controller method naming** — `AttendanceController` has both clock-in and clock-out methods named `getAttendanceRecord`. To be renamed during refactoring.
